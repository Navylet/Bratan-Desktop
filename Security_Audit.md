# Security Audit Report
## OpenClaw Desktop (Братан Desktop)

**Дата:** 2026-03-07  
**Аудитор:** Electron Architect  
**Область:** Main process, renderer process, IPC, интеграции

---

## 1. Обзор безопасности

Приложение работает в среде Electron, где традиционные веб‑угрозы сочетаются с доступом к Node.js API. Текущая конфигурация содержит **критические уязвимости**, требующие немедленного исправления.

---

## 2. Критические уязвимости (P0)

### 2.1. Отключённая контекстная изоляция
**Файл:** main.js:26  
**Описание:** `contextIsolation: false` позволяет renderer‑процессу напрямую обращаться к Node.js API через глобальный объект, даже если `nodeIntegration` выключена (но в данном случае она включена).  
**Угроза:** XSS‑атака может выполнить произвольный код на стороне системы.  
**Эксплуатация:** Если злонамеренный скрипт попадает в renderer (через инжектированный контент или загрузку внешней страницы), он получает полный доступ к файловой системе, процессам, сетям.  
**Исправление:** Установить `contextIsolation: true` и убедиться, что preload.js использует `contextBridge` для предоставления только необходимых API.

### 2.2. Включённая nodeIntegration
**Файл:** main.js:25  
**Описание:** `nodeIntegration: true` даёт renderer‑процессу доступ к модулям Node.js через `require`.  
**Угроза:** Усиливает предыдущую уязвимость, делает возможным прямой `require('child_process')` и выполнение команд.  
**Исправление:** Установить `nodeIntegration: false`. Все Node.js операции должны выполняться в main process и вызываться через IPC.

### 2.3. Отсутствие Content Security Policy (CSP)
**Файл:** index.html  
**Описание:** Нет ограничений на загрузку и выполнение скриптов.  
**Угроза:** Инжектированный скрипт (через уязвимость в коде renderer) выполнится без препятствий.  
**Исправление:** Добавить строгий CSP meta‑тег, разрешающий только доверенные источники.

### 2.4. Включённый enableRemoteModule
**Файл:** main.js:27  
**Описание:** `enableRemoteModule: true` включает устаревший модуль `remote`, который позволяет renderer вызывать main‑процесс методы.  
**Угроза:** Обход IPC и контекстной изоляции. Модуль имеет известные уязвимости.  
**Исправление:** Удалить эту опцию (по умолчанию false в Electron >10).

### 2.5. Отсутствие sandbox
**Файл:** main.js (webPreferences)  
**Описание:** Sandbox не включён, поэтому renderer‑процесс работает с привилегиями пользователя.  
**Угроза:** Уязвимости в Chromium могут привести к компрометации системы.  
**Исправление:** Добавить `sandbox: true`. При необходимости использовать preload для безопасных API.

## 3. Серьёзные уязвимости (P1)

### 3.1. Отсутствие валидации входных данных IPC
**Файлы:** main.js (обработчики ipcMain), preload.js  
**Описание:** IPC‑методы принимают произвольные строки (пути к файлам, команды) без проверки.  
**Угроза:** Path traversal, инъекция команд, доступ к файлам за пределами рабочей директории.  
**Пример:** `window.api.fs.readFile('../../../etc/passwd')` может прочитать системный файл.  
**Исправление:** Валидировать все входные параметры: проверять, что путь находится внутри разрешённой директории, экранировать специальные символы.

### 3.2. Хранение токенов в plain text
**Файлы:** integrations/google.js, integrations/github.js  
**Описание:** Токены OAuth сохраняются в локальном хранилище без шифрования.  
**Угроза:** Если злоумышленник получит доступ к файловой системе, он может украсть токены.  
**Исправление:** Использовать `safeStorage` API Electron (только macOS/Linux) или шифровать с помощью ключа, хранимого в системном keychain (keytar).

### 3.3. Отсутствие проверки источников IPC‑сообщений
**Файл:** main.js  
**Описание:** Обработчики ipcMain не проверяют, от какого окна/процесса пришло сообщение.  
**Угроза:** Если открыть второе окно (например, через DevTools), оно может отправлять команды в main process.  
**Исправление:** Использовать `event.sender` для проверки origin или добавить канал‑специфичные проверки.

### 3.4. Загрузка внешних ресурсов без integrity check
**Файл:** index.html (Tailwind CSS, Font Awesome, Google Fonts)  
**Описание:** CDN‑ресурсы загружаются без SRI (Subresource Integrity).  
**Угроза:** Если CDN скомпрометирован, злонамеренный код будет выполнен в приложении.  
**Исправление:** Добавить атрибут `integrity` к script/link тегам или загружать ресурсы локально.

## 4. Уязвимости средней тяжести (P2)

### 4.1. Отсутствие защиты от brute force (интеграции)
**Файлы:** integrations/google.js, integrations/github.js  
**Описание:** Нет ограничения попыток аутентификации.  
**Угроза:** Злоумышленник может пытаться угадать токен или подобрать пароль (если используется базовая аутентификация).  
**Исправление:** Добавить задержки после неудачных попыток, ограничить количество запросов в минуту.

### 4.2. Debug‑конфигурация в production
**Файл:** dist/builder-debug.yml  
**Описание:** Используется debug‑конфигурация сборщика, которая может включать отладочные опции.  
**Угроза:** Упрощает реверс‑инжиниринг, может раскрывать внутреннюю структуру.  
**Исправление:** Использовать отдельную production‑конфигурацию с минификацией и без source maps.

### 4.3. Отсутствие контроля версий зависимостей
**Файл:** package.json  
**Описание:** Некоторые зависимости зафиксированы на конкретных версиях, но нет регулярного обновления.  
**Угроза:** Известные уязвимости в старых версиях не исправляются.  
**Исправление:** Внедрить регулярное обновление зависимостей (например, с помощью Dependabot).

### 4.4. Логирование чувствительной информации
**Файлы:** renderer.js, main.js  
**Описание:** В логи могут попадать токены, пути к файлам, личные данные.  
**Угроза:** Если логи сохраняются в файл, они могут быть прочитаны.  
**Исправление:** Маскировать чувствительные данные перед логированием.

## 5. Рекомендации по устранению

### 5.1. Немедленные действия (1 день)

1. **Изменить webPreferences в main.js:**
   ```javascript
   webPreferences: {
     nodeIntegration: false,
     contextIsolation: true,
     sandbox: true,
     preload: path.join(__dirname, 'preload.js')
   }
   ```

2. **Добавить CSP в index.html:**
   ```html
   <meta http-equiv="Content-Security-Policy" content="
     default-src 'self';
     script-src 'self' https://cdn.tailwindcss.com https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/;
     style-src 'self' 'unsafe-inline' https://cdn.tailwindcss.com https://fonts.googleapis.com;
     font-src 'self' https://fonts.gstatic.com;
     img-src 'self' data:;
   ">
   ```

3. **Удалить `enableRemoteModule: true`.**

4. **Протестировать приложение после изменений.**

### 5.2. Краткосрочные действия (1‑2 недели)

5. **Валидация входных данных IPC:**
   Создать модуль-валидатор для путей, URL, команд.

6. **Шифрование токенов:**
   Использовать `safeStorage` или keytar.

7. **Проверка источников IPC:**
   Добавить `validateSender(event.sender)`.

8. **Добавить SRI для CDN‑ресурсов** или перейти на локальные ресурсы.

### 5.3. Долгосрочные улучшения (1 месяц)

9. **Внедрить мониторинг безопасности:**
   Логировать подозрительные IPC‑вызовы, failed authentication.

10. **Регулярное обновление зависимостей.**

11. **Провести penetration testing** с помощью инструментов типа Electronegativity.

12. **Добавить двухфакторную аутентификацию** для критичных интеграций.

---

## 6. Проверка после исправлений

Рекомендуется проверить приложение с помощью следующих инструментов:

- **Electronegativity:** статический анализ конфигурации Electron.
- **Chrome DevTools Security panel:** проверка CSP, isolation.
- **Node Security Platform (npm audit):** уязвимости в зависимостях.

---

## 7. Заключение

Текущее состояние безопасности OpenClaw Desktop **не соответствует минимальным требованиям** для десктопного приложения с доступом к пользовательским данным. Критические уязвимости позволяют скомпрометировать систему через XSS или инъекцию.

Рекомендуется **немедленно приступить к исправлению P0‑уязвимостей**, после чего реализовать P1 и P2 улучшения. Безопасность должна быть приоритетом на всех этапах разработки.
