# Рекомендации по улучшению QA‑процесса и автоматизации тестирования

## Текущее состояние
- Тестирование ручное, не систематическое.
- Нет unit‑, integration‑, e2e‑тестов.
- Нет CI/CD pipeline с запуском тестов.
- Баги фиксируются ad‑hoc, нет централизованного трекинга.

## Цель
Построить зрелый QA‑процесс, который обеспечивает высокое качество каждой сборки, быстрое выявление регрессий и уверенность при развёртывании на всех трёх платформах.

## Рекомендации

### 1. Внедрить автоматические тесты
#### 1.1 Unit‑тесты (Jest)
- **Что тестировать:** Отдельные функции в `renderer.js`, `main.js`, модули интеграций.
- **Инструменты:** Jest + `ts‑jest` (если будет TypeScript), `electron‑mocks`.
- **Примеры:**
  - Функция `addMessage` (рендер сообщения).
  - Фильтрация логов по уровню.
  - Обработчики IPC в main (проверка корректности вызовов).
- **Покрытие:** Стремиться к 80%+ покрытия кода.

#### 1.2 Integration‑тесты (Playwright / Spectron)
- **Что тестировать:** Взаимодействие между фронтендом и бэкендом (IPC), интеграции с внешними сервисами (моки).
- **Инструменты:** Playwright (рекомендуется) или Spectron (устаревший). Playwright поддерживает Electron, имеет встроенные моки сети.
- **Примеры:**
  - Отправка сообщения в чат → получение ответа от агента.
  - Загрузка списка файлов из Google Drive (мок API).
  - Сохранение файла через редактор.
- **Преимущества:** Более надёжно, чем unit‑тесты, охватывают ключевые пользовательские сценарии.

#### 1.3 E2E‑тесты (Playwright)
- **Что тестировать:** Полные пользовательские сценарии, кросс‑платформенное поведение.
- **Инструменты:** Playwright для Electron.
- **Примеры:**
  - Пользователь открывает приложение, переходит во вкладку «Файлы», создаёт папку, открывает файл, редактирует, сохраняет.
  - Пользователь настраивает интеграцию GitHub, загружает список репозиториев.
  - Пользователь запускает агента, отслеживает его статус, останавливает.
- **Запуск:** В CI на всех трёх платформах (Windows, macOS, Linux).

#### 1.4 Моки и изоляция
- **Моки для внешних сервисов:** Использовать `MSW` (Mock Service Worker) или встроенные моки Playwright для Google/GitHub/Perplexity API.
- **Моки для файловой системы:** `memfs` или `mock‑fs` для тестирования операций с файлами без реального доступа к диску.
- **Моки для IPC:** `jest.mock` для `window.api` в renderer‑тестах.

### 2. Настроить CI/CD pipeline
- **Платформа:** GitHub Actions (бесплатно для open‑source) или GitLab CI.
- **Этапы:**
  1. **Install:** Установка зависимостей (Node.js, системные библиотеки).
  2. **Lint:** Запуск линтера (ESLint) и форматтера (Prettier).
  3. **Unit tests:** Запуск Jest.
  4. **Integration tests:** Запуск Playwright в headless‑режиме.
  5. **Build:** Сборка приложения под все три платформы (electron‑builder).
  6. **E2E tests:** Запуск Playwright на собранном приложении (можно на Linux с xvfb).
  7. **Deploy:** Автоматическая публикация релизов (GitHub Releases).

- **Кросс‑платформенная сборка:** Использовать матрицу в GitHub Actions:
  ```yaml
  strategy:
    matrix:
      os: [ubuntu‑latest, windows‑latest, macos‑latest]
  ```

### 3. Внедрить статический анализ
- **ESLint:** Проверка стиля кода, поиск потенциальных ошибок (правила `eslint‑recommended`, `plugin:security/recommended`).
- **TypeScript:** Миграция на TypeScript для повышения надёжности и автодополнения.
- **Security audit:** `npm audit`, `snyk` для проверки уязвимостей в зависимостях.

### 4. Централизованный баг‑трекинг
- **Инструмент:** GitHub Issues (бесплатно и интегрируется с репозиторием).
- **Процесс:**
  1. Тестировщик создаёт issue с шаблоном (environment, steps, expected/actual, screenshots).
  2. Автоматически назначается метка `bug`.
  3. Разработчик исправляет, ссылается на pull request.
  4. После мержа PR issue закрывается.
- **Метки:** `critical`, `major`, `minor`, `trivial`, `platform‑windows`, `platform‑macos`, `platform‑linux`.

### 5. Регрессионное тестирование
- **Набор регрессионных тестов:** Ключевые тест‑кейсы из TEST_PLAN.md автоматизировать (Playwright).
- **Запуск перед каждым релизом:** Вручную или автоматически в CI.
- **Визуальное регрессионное тестирование:** Использовать `playwright‑screenshot` для сравнения скриншотов критических интерфейсов.

### 6. Кросс‑платформенное тестирование на реальном железе
- **Вирутальные машины:** Использовать VirtualBox/VMware с образами Windows/macOS для периодического ручного тестирования.
- **Облачные сервисы:** `BrowserStack` или `Sauce Labs` предоставляют реальные машины с macOS/Windows для тестирования Electron.
- **Хакинтош:** Если нет доступа к Mac, можно рассмотреть хакинтош (не для продакшена, только для тестирования).

### 7. Документация тестирования
- **Тест‑план:** Держать TEST_PLAN.md в актуальном состоянии.
- **Чек‑листы:** Smoke‑test чек‑лист для быстрой проверки после сборки.
- **Руководство по тестированию интеграций:** Как настроить тестовые аккаунты Google/GitHub/Perplexity, какие сценарии проверять.

### 8. Мета‑тестирование (QA процесса)
- **Регулярные ретроспективы:** Что сломалось в релизе, почему не было поймано тестами.
- **Обновление тестов:** При добавлении новой функциональности сразу писать тесты (TDD/BDD).
- **Метрики качества:** Отслеживать количество багов в релизе, время на их исправление, покрытие тестами.

## Пример конфигурации (кратко)

### package.json scripts
```json
{
  "scripts": {
    "test:unit": "jest",
    "test:integration": "playwright test --project=electron",
    "test:e2e": "playwright test --project=electron‑packaged",
    "lint": "eslint .",
    "build": "electron‑builder",
    "ci": "npm run lint && npm run test:unit && npm run build && npm run test:e2e"
  }
}
```

### playwright.config.js
```javascript
module.exports = {
  projects: [
    {
      name: 'electron',
      use: { device: 'Desktop Chrome' },
      testDir: './integration‑tests',
    },
    {
      name: 'electron‑packaged',
      use: { device: 'Desktop Chrome' },
      testDir: './e2e‑tests',
      launchOptions: { executablePath: './dist/OpenClaw‑linux‑x64/openclaw‑desktop' }
    }
  ]
};
```

## Приоритизация внедрения
1. **Срочно:** Настроить ESLint и Prettier, добавить unit‑тесты для критических функций (обработка ошибок).
2. **Ближайший релиз:** Внедрить Playwright integration‑тесты для основных сценариев (чат, файлы, агенты).
3. **Следующий квартал:** Настроить CI/CD с кросс‑платформенной сборкой и e2e‑тестами.
4. **Долгосрочно:** Миграция на TypeScript, внедрение визуального регрессионного тестирования.

## Ожидаемый эффект
- Снижение количества багов в продакшене на 70‑80%.
- Уверенность при развёртывании на всех платформах.
- Ускорение разработки за счёт автоматической проверки регрессий.
- Профессиональный имидж проекта.

**QA — это не отдельная фаза, а интегральная часть разработки.** Внедрение этих практиций превратит OpenClaw Desktop в надёжный, профессиональный продукт.
