# План действий: Исправление сборки OpenClaw Desktop

## Цель
Устранить ошибки конфигурации electron‑builder, обеспечить рабочую сборку для Windows, macOS и Linux.

## Общая оценка
- **Сроки**: 1–2 рабочих дня (включая тестирование).
- **Риски**: Возможна несовместимость с некоторыми платформами, необходимость дополнительных настроек для подписи и нотаризации.
- **Ресурсы**: Разработчик с опытом работы с Electron и electron‑builder.

## Шаги

### Этап 1: Подготовка (1 час)
1. **Создать резервную копию текущих конфигурационных файлов**
   - Скопировать `package.json`, `electron-builder.json`, `build/` в отдельную папку.
2. **Обновить зависимости**
   - Обновить `electron-builder` до последней стабильной версии (26.8.1) в `package.json`.
   - Добавить недостающий пакет `electron-notarize` в `devDependencies`.
   - Выполнить `npm install` или `yarn install`.
3. **Проверить версию Electron**
   - Убедиться, что версия Electron совместима с electron‑builder 26.x (Electron 40.8.0 – совместима).

### Этап 2: Исправление конфигурации (2–3 часа)
1. **Убрать поле `build` из `package.json`**
   - Удалить секцию `"build": { "extends": "./electron-builder.json" }`.
   - Оставить поле `"main": "main.js"` (оно нужно для Electron, но не для electron‑builder).
2. **Переработать `electron-builder.json`**
   - Привести файл в соответствие со схемой electron‑builder 26.x.
   - Удалить неизвестные свойства (`main`, `linux.desktop`, `mac.notarize`, лишние `win`-поля).
   - Перенести необходимые настройки в правильные секции.
   - Пример структуры (на основе документации):
     ```json
     {
       "appId": "com.openclaw.desktop",
       "productName": "OpenClaw Desktop",
       "directories": {
         "output": "dist"
       },
       "files": [
         "main.js",
         "renderer/**/*",
         "integrations/**/*",
         "node_modules/**/*"
       ],
       "linux": {
         "target": ["AppImage", "deb", "rpm"],
         "icon": "build/icons/",
         "category": "Utility"
       },
       "mac": {
         "target": ["dmg", "zip"],
         "icon": "build/icons/",
         "hardenedRuntime": true,
         "gatekeeperAssess": false,
         "notarize": false
       },
       "win": {
         "target": ["nsis", "portable"],
         "icon": "build/icons/"
       }
     }
     ```
   - При необходимости добавить конфигурацию для подписи и нотаризации (см. этап 4).
3. **Проверить валидность конфигурации**
   - Запустить `npx electron-builder --dir` для проверки конфигурации без сборки.
   - Убедиться, что ошибки не появляются.

### Этап 3: Сборка и тестирование (3–4 часа)
1. **Локальная сборка для текущей ОС (Linux)**
   - Выполнить `npm run dist:linux` (или соответствующий скрипт).
   - Проверить, что приложение запускается, все вкладки и интеграции работают.
2. **Сборка для Windows (на Linux через Wine)**
   - Установить Wine и необходимые зависимости.
   - Запустить `npm run dist:win`.
   - Протестировать полученный `.exe` в виртуальной машине или через Wine.
3. **Сборка для macOS (на Linux через tools)**
   - Если нет доступа к macOS, использовать Docker‑образ `electron‑userland/builder:latest` или подобный.
   - Запустить `npm run dist:mac`.
   - Проверить сборку на реальном Mac или эмуляторе.

### Этап 4: Настройка подписи и нотаризации (опционально, 1–2 дня)
1. **Подпись для Windows**
   - При наличии сертификата добавить в конфигурацию:
     ```json
     "win": {
       "certificateFile": "path/to/cert.pfx",
       "certificatePassword": "",
       "signingHashAlgorithms": ["sha256"],
       "rfc3161TimeStampServer": "http://timestamp.digicert.com"
     }
     ```
2. **Нотаризация для macOS**
   - Установить `electron-notarize`.
   - Настроить скрипт нотаризации в `afterSign` хуке.
   - Добавить в конфигурацию:
     ```json
     "mac": {
       "notarize": true,
       "hardenedRuntime": true
     }
     ```
   - Указать Apple ID и пароль в переменных окружения.

### Этап 5: Автоматизация и документация (1 час)
1. **Обновить скрипты сборки в `package.json`**
   - Проверить, что `dist:win`, `dist:mac`, `dist:linux` работают корректно.
   - Добавить скрипт `dist:all` для сборки всех платформ.
2. **Задокументировать процесс сборки**
   - Создать `BUILD.md` с инструкциями для разработчиков.
   - Описать требования к окружению для сборки под разные ОС.
3. **Настроить CI/CD (если требуется)**
   - Интегрировать сборку в GitHub Actions / GitLab CI.
   - Автоматизировать создание релизов.

## Риски и меры их снижения
- **Риск**: Сборка для Windows/macOS не работает из‑за отсутствия подписи.
  - **Мера**: Сначала собрать неподписанные версии для тестирования функциональности.
- **Риск**: Конфигурация по‑прежнему содержит невалидные поля.
  - **Мера**: Использовать `electron‑builder --help` и официальную документацию для проверки схемы.
- **Риск**: Интеграции перестают работать из‑за изменений в структуре проекта.
  - **Мера**: Провести тщательное тестирование каждой интеграции после сборки.
- **Риск**: Увеличение размера бандла из‑за лишних файлов.
  - **Мера**: Настроить `files` в конфигурации, исключить ненужные ресурсы.

## Критерии успеха
1. Сборка завершается без ошибок для всех трёх платформ.
2. Собранные приложения запускаются и все основные функции работают (вкладки, интеграции).
3. Размер дистрибутивов соответствует ожиданиям (для Electron).
4. Процесс сборки задокументирован и может быть выполнен новым разработчиком.
