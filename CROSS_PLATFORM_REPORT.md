# Отчёт по кросс‑платформенной совместимости OpenClaw Desktop

## Цель
Оценить готовность Electron‑приложения к работе на трёх целевых платформах: **Windows 10/11**, **macOS (latest)**, **Linux (Ubuntu/Debian/Fedora)**. Выявить потенциальные проблемы, связанные с различиями ОС.

## Методология
- Статический анализ кода (main.js, preload.js, renderer.js, package.json)
- Проверка использования платформозависимых API (файловые пути, команды shell, GUI)
- Проверка зависимостей (библиотеки, нативные модули)
- Тестирование на Linux (WSL2) с ограниченным GUI

## Результаты по платформам

### Linux (Ubuntu/Debian/Fedora)
**Статус:** Частично работает (с ограничениями)

**Проблемы:**
1. **Зависимости GUI:** Electron требует библиотеки `libnspr4`, `libnss3`, которые могут отсутствовать в минимальных установках (особенно в контейнерах). В WSL2 без GUI приложение не запускается.
2. **Пути к CLI:** Команда `openclaw` предполагает установку CLI в PATH. Если CLI установлен глобально (`npm i -g openclaw`), проблем нет.
3. **Файловая система:** Используются стандартные Node.js `fs` API, проблем нет.
4. **Интеграции:** Google OAuth требует браузера по умолчанию. В headless‑окружении может не сработать.

**Рекомендации для Linux:**
- Добавить в документацию список системных зависимостей (`libnspr4`, `libnss3`, `libxss1`, `libgconf-2-4` и т.д.).
- Предусмотреть fallback для headless‑окружений (например, вывод ссылки для копирования).
- Проверить работу в разных дистрибутивах (особенно с Wayland vs X11).

### Windows 10/11
**Статус:** Предполагаемые проблемы (не тестировалось)

**Проблемы:**
1. **Пути к CLI:** В коде используется команда `openclaw` без расширения `.exe`. На Windows исполняемый файл обычно называется `openclaw.exe`. Без учёта этого запуск Gateway не сработает.
2. **Файловые пути:** В некоторых местах используются Unix‑стиль разделителей (`/`). Node.js `path` модуль должен нивелировать это, но если есть конкатенация строк, могут возникнуть ошибки.
3. **Автозапуск:** Реализация автозапуска приложения при старте системы отличается от Linux (реестр vs systemd).
4. **Системный трей:** В Windows иконка в трее может вести себя иначе (клик правой кнопкой, меню). Нужно проверить работу `Tray` API.
5. **Уведомления:** Electron `Notification` API использует системные уведомления; на Windows они могут выглядеть по‑другому, но функциональность сохраняется.
6. **Прокси:** Корпоративные сети часто требуют прокси; в Windows настройки прокси берутся из системы, но интеграции могут их игнорировать.

**Рекомендации для Windows:**
- Использовать `process.platform` для определения команды CLI (`openclaw.exe`).
- Протестировать сборку `win-unpacked` из директории `dist`.
- Проверить работу с длинными путями (более 260 символов) — может потребоваться включение поддержки длинных путей в манифесте.
- Проверить автозапуск через реестр (`app.setLoginItemSettings`).

### macOS (latest)
**Статус:** Предполагаемые проблемы (не тестировалось)

**Проблемы:**
1. **Подписка и Gatekeeper:** Неподписанное приложение может быть заблокировано Gatekeeper. Пользователю придётся обходить предупреждения (правая кнопка → «Открыть»).
2. **Меню приложения:** Electron по умолчанию создаёт стандартное меню macOS (верхняя панель). Нужно убедиться, что оно корректно (например, есть пункт «About», «Quit»).
3. **Горячие клавиши:** Сочетания типа `Cmd+Q` (выход) и `Cmd+W` (закрыть окно) должны работать ожидаемо. В коде могут быть конфликты.
4. **Пути к CLI:** Предполагается, что `openclaw` установлен глобально через `npm`. На macOS это обычно `/usr/local/bin/openclaw`. Если установка через `brew`, путь может отличаться.
5. **Интеграция с Finder:** Открытие внешней папки (`shell.openPath`) использует Finder, проблем нет.
6. **Иконки в доке:** Нужно проверить, что иконка приложения отображается корректно, нет артефактов.

**Рекомендации для macOS:**
- Собрать `.dmg` или `.zip` с подписью (если есть Developer ID).
- Протестировать поведение меню (можно кастомизировать через `Menu.setApplicationMenu`).
- Проверить работу с разрешениями Privacy (камера, микрофон, файлы) — если приложение запрашивает доступ к файлам, нужно добавить соответствующие описания в `Info.plist`.

## Критические кросс‑платформенные проблемы

| Проблема | Затронутые ОС | Severity | Описание |
|----------|---------------|----------|----------|
| Отсутствие модулей `googleIntegration.js`, `githubIntegration.js` | Все | Critical | Модули не существуют, приложение не запускается. |
| CLI команда `openclaw` без учёта расширения .exe | Windows | Major | На Windows команда должна быть `openclaw.exe`. |
| Нет проверки запущенного Gateway | Все | Minor | Возможен конфликт портов при повторном запуске. |
| Отсутствие индикаторов загрузки | Все | Minor | Плохой UX на всех платформах. |
| Необработанные ошибки промисов | Все | Major | Приложение может «зависать» без обратной связи. |
| Нет поддержки прокси | Все | Minor | В корпоративных сетях интеграции не работают. |

## Рекомендации по унификации

1. **Использовать `path.join()` и `path.sep`** для всех операций с путями.
2. **Определять платформу** через `process.platform` и ветвить логику для CLI, путей, меню.
3. **Добавить проверку зависимостей** при запуске (например, через `which openclaw`).
4. **Интегрировать настройки прокси** через `electron.net` или глобальные переменные окружения.
5. **Протестировать на реальных машинах/VM** каждую платформу перед релизом.

## Инструменты для кросс‑платформенного тестирования
- **VM:** VirtualBox/VMware с образами Windows/macOS.
- **CI/CD:** GitHub Actions (windows‑latest, macos‑latest, ubuntu‑latest) для автоматического запуска тестов.
- **Эмуляторы:** `electron‑packager`/`electron‑builder` для сборки под все ОС.
- **Средства проверки:** `spectron` (e2e), `jest` (unit), `playwright` (интеграционные).

## Вывод
