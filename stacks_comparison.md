# Сравнительный анализ альтернативных стеков для OpenClaw Desktop

## Критерии оценки
1. **Время и сложность реализации** – оценка усилий для достижения рабочего состояния.
2. **Производительность** – время загрузки, использование памяти, отзывчивость.
3. **Безопасность** – риски уязвимостей, изоляция кода, доступ к системе.
4. **Кросс‑платформенность** – поддержка Windows, macOS, Linux.
5. **Поддержка текущих интеграций** – совместимость с OpenClaw JS, Google APIs, GitHub, Perplexity API.
6. **Сохранение текущего кода фронтенда** – возможность использования существующего React + TypeScript кода.
7. **Сложность миграции** – объём переписывания, необходимость изучения новых технологий.

## Варианты

### 1. Electron (исправленная конфигурация)
- **Описание**: Оставить текущий стек, обновить electron‑builder, исправить конфигурацию.
- **Время и сложность**: Низкая (2–4 часа). Требуется обновление зависимостей и правка конфигурационных файлов.
- **Производительность**: Средняя. Electron приносит накладные расходы Chromium и Node.js (~100–300 МБ RAM, запуск 1–3 сек).
- **Безопасность**: Средняя. Полный доступ к Node.js, что может быть риском, если приложение работает с конфиденциальными данными.
- **Кросс‑платформенность**: Отличная (Windows, macOS, Linux). Поддержка всех трёх платформ из коробки.
- **Поддержка текущих интеграций**: Полная. Все интеграции используют Node.js API и работают без изменений.
- **Сохранение фронтенда**: 100% сохранение кода React + TypeScript.
- **Сложность миграции**: Минимальная (только исправление сборки).

### 2. Tauri (Rust + WebView)
- **Описание**: Заменить Electron на Tauri: Rust‑бэкенд + системный WebView (WebKit на macOS, WebView2 на Windows, WebKitGTK на Linux).
- **Время и сложность**: Высокая (2–4 недели). Требуется написание бэкенда на Rust, перенос интеграций, настройка сборки.
- **Производительность**: Высокая. Минимальный размер бандла (<10 МБ), низкое потребление памяти (только WebView), быстрая загрузка.
- **Безопасность**: Высокая. Изоляция кода, ограниченный доступ к системе через явные IPC‑вызовы.
- **Кросс‑платформенность**: Хорошая (Windows, macOS, Linux). Требует установки WebView2 на Windows (может быть поставлен вместе с приложением).
- **Поддержка текущих интеграций**: Частичная. Интеграции, требующие Node.js (OpenClaw JS, Google APIs), нужно переписать на Rust или использовать через REST API.
- **Сохранение фронтенда**: Полное (React + TypeScript работает в WebView).
- **Сложность миграции**: Высокая. Необходимо изучить Rust, переписать бэкенд, адаптировать IPC.

### 3. NW.js (Node.js + Chromium)
- **Описание**: Альтернатива Electron, где Node.js и Chromium работают в одном процессе.
- **Время и сложность**: Средняя (1–2 недели). Аналогичен Electron, но требует замены API и пересборки.
- **Производительность**: Средняя (схожа с Electron). Накладные расходы Chromium и Node.js.
- **Безопасность**: Средняя (как у Electron).
- **Кросс‑платформенность**: Отличная (Windows, macOS, Linux).
- **Поддержка текущих интеграций**: Полная. Node.js API доступен.
- **Сохранение фронтенда**: Полное. React + TypeScript работает без изменений.
- **Сложность миграции**: Средняя. Нужно заменить Electron‑специфичные API на NW.js API (например, `nw.Window` вместо `BrowserWindow`).

### 4. Neutralino.js (WebView + нейтивный бэкенд)
- **Описание**: Лёгкий фреймворк, использующий системный WebView и небольшой нейтивный бэкенд (C++, Python, Go).
- **Время и сложность**: Средняя (1–2 недели). Требуется переписать бэкенд на поддерживаемом языке, настроить IPC.
- **Производительность**: Высокая (малый размер, низкое потребление памяти).
- **Безопасность**: Высокая (изоляция, ограниченный API).
- **Кросс‑платформенность**: Хорошая (Windows, macOS, Linux).
- **Поддержка текущих интеграций**: Низкая. Node.js недоступен, интеграции нужно переписывать на других языках.
- **Сохранение фронтенда**: Полное (WebView).
- **Сложность миграции**: Средняя/высокая. Необходимо изучить новый бэкенд и IPC.

### 5. Flutter Desktop (Dart + Flutter)
- **Описание**: Полный переход на Flutter, включая UI и логику.
- **Время и сложность**: Очень высокая (1–3 месяца). Полная переработка интерфейса и логики.
- **Производительность**: Высокая (компилированный код, нативный рендеринг).
- **Безопасность**: Высокая (изоляция, безопасный язык).
- **Кросс‑платформенность**: Отличная (Windows, macOS, Linux).
- **Поддержка текущих интеграций**: Очень низкая. Большинство интеграций нужно переписать с использованием Dart‑пакетов (если они существуют).
- **Сохранение фронтенда**: Нулевое. Весь фронтенд переписывается на Dart/Flutter.
- **Сложность миграции**: Очень высокая. Требуется освоение Flutter и переписывание всей кодовой базы.

### 6. Wails (Go + WebView)
- **Описание**: Go‑бэкенд + системный WebView, аналогично Tauri, но с Go вместо Rust.
- **Время и сложность**: Средняя (2–3 недели). Написание бэкенда на Go, перенос интеграций.
- **Производительность**: Высокая (малый размер, низкое потребление памяти).
- **Безопасность**: Высокая (изоляция, ограниченный IPC).
- **Кросс‑платформенность**: Хорошая (Windows, macOS, Linux).
- **Поддержка текущих интеграций**: Частичная. Node.js интеграции нужно переписать на Go.
- **Сохранение фронтенда**: Полное (WebView).
- **Сложность миграции**: Средняя. Go проще Rust, но всё равно требуется переписать бэкенд.

### 7. Qt (C++/QML)
- **Описание**: Нативное приложение на Qt с QML для интерфейса.
- **Время и сложность**: Очень высокая (3–6 месяцев). Полная переработка на C++/QML.
- **Производительность**: Очень высокая (нативное исполнение).
- **Безопасность**: Высокая (память управляется вручную, но можно допустить уязвимости).
- **Кросс‑платформенность**: Отличная (Windows, macOS, Linux).
- **Поддержка текущих интеграций**: Очень низкая. Все интеграции нужно переписать на C++.
- **Сохранение фронтенда**: Нулевое.
- **Сложность миграции**: Очень высокая. Требует экспертизы C++ и Qt.
